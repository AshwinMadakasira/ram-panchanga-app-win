# Windows Desktop Guide

This guide explains how the RAM Panchanga Windows app is structured, how to work on it locally, and what still matters for production distribution.

## What This Repo Is

This repo is the Windows desktop distribution of RAM Panchanga.

It combines:

- an Expo + React Native app
- a React Native Web export path
- a Tauri native desktop shell

The frontend and domain logic live in the normal app code. Tauri is only the native host and packaging layer.

## High-Level Architecture

The app is built from two cooperating parts:

1. App/frontend layer
   Lives in [app](../app) and [src](../src)
2. Native desktop layer
   Lives in [src-tauri](../src-tauri)

The important runtime flow is:

1. Tauri opens a window.
2. In development, Tauri points to `http://localhost:1420`.
3. In release, Tauri loads the exported files from `dist`.
4. The React app bootstraps fonts and SQLite.
5. The bundled seed data is imported into local SQLite.
6. Screens read from repositories through hooks.

## Local Setup

### Required tools

- Node.js and npm
- Rust via `rustup`
- Visual Studio Build Tools with MSVC and Windows SDK
- WebView2 runtime

### Install dependencies

```bash
npm install
```

### Verify the machine

```bash
npx tauri info
```

If this fails, fix the local toolchain before trying to run or build the app.

## Development Workflow

### Run the desktop app

```bash
npm run desktop:dev
```

What happens:

- Expo starts a web dev server on port `1420`
- Tauri starts the Rust app
- the desktop window loads the app from the dev server

### Export the web bundle

```bash
npm run desktop:bundle
```

This writes the production frontend bundle into `dist`.

### Build Windows installers

```bash
npm run desktop:build
```

This:

- exports the web app
- compiles the Tauri/Rust shell
- creates Windows release artifacts

The final installers land under:

- `src-tauri/target/release/bundle/msi`
- `src-tauri/target/release/bundle/nsis`

## Validation Workflow

Run these commands for routine validation:

```bash
npm run typecheck
npm run lint
npm run test
npm run test:desktop-smoke
```

What each one covers:

- `typecheck`
  TypeScript compile safety
- `lint`
  Static code issues in the app-owned source
- `test`
  Seed data and domain sanity checks
- `test:desktop-smoke`
  Desktop/web regression check for blank-screen style failures

## Desktop-Specific Changes In This Repo

These are the main desktop adaptations already made:

### Tauri shell added

- Native app configuration and icons were added in [src-tauri](../src-tauri)

### Metro adjusted for desktop export

- [metro.config.js](../metro.config.js) was updated so Expo web includes the SQLite wasm asset
- Rust build output is blocked from Metro watching

### Reminder startup guarded on desktop

- [src/components/app/ReminderCoordinator.tsx](../src/components/app/ReminderCoordinator.tsx)
- [src/services/reminders.ts](../src/services/reminders.ts)

These avoid boot-time problems from mobile-only notification APIs on web/desktop.

### Zustand web workaround

- [src/store/settings-store.ts](../src/store/settings-store.ts)
- [src/types/zustand-cjs.d.ts](../src/types/zustand-cjs.d.ts)

These avoid a Zustand package-export issue that caused blank-screen failures in the desktop build path.

## Known Limitations

### Desktop reminders are not implemented

The mobile reminder flow is intentionally disabled on web/desktop. The app runs, but desktop-native scheduling is not present yet.

### WebView2 is still a target-machine dependency

The packaged app relies on the system WebView runtime on Windows.

### Code signing is still required for smooth public distribution

The generated installers are not signed by default. They will commonly trigger Windows warnings until they are signed.

### Generated build output is large

`src-tauri/target` and `dist` are build outputs, not source. They should not be committed as normal source artifacts.
